Lecture 01 · The Body

Foundation — Setup & Python Essentials

The architecture of contemporary Internet of Things (IoT) robotics relies heavily on powerful, low-cost microcontrollers capable of multitasking and wireless communication. The ESP32 serves as the foundational hardware for this curriculum. Manufactured by Espressif Systems, the ESP32 is a premier System-on-a-Chip (SoC) featuring a dual-core Xtensa 32-bit LX6 microprocessor operating at up to 240 MHz. This architecture is augmented by integrated 802.11 b/g/n Wi-Fi and Bluetooth v4.2 BR/EDR and BLE, making it an ideal candidate for cloud-connected robotics.

Historically, microcontrollers of this class were programmed strictly in compiled, low-level languages like C or C++ (e.g., using the Arduino IDE or ESP-IDF). While C++ offers bare-metal execution speed and granular memory control, it introduces significant friction in the prototyping phase due to complex build toolchains, manual pointer management, and lengthy compile-flash-test cycles. The ESP32, however, possesses sufficient flash memory and Random Access Memory (RAM) to host a language interpreter — specifically, MicroPython.

The Python Paradigm in Embedded Systems

Python is an interpreted, high-level, dynamically typed language. In traditional desktop or server environments, Python abstracts memory management, garbage collection, and hardware interactions entirely away from the developer. MicroPython is a lean, highly efficient implementation of Python 3 designed explicitly to run on resource-constrained microcontrollers. It maintains the core syntax, semantics, and object-oriented capabilities of Python but replaces heavy standard libraries with hardware-specific modules, such as machine and network.

The primary advantage of deploying Python in a robotics context is state management and rapid iteration. MicroPython operates via a Read-Eval-Print Loop (REPL), which allows developers to execute Python code line-by-line directly on the microcontroller over a serial connection. This dynamic execution environment permits real-time testing of motor drivers and sensors without recompiling the entire codebase. Furthermore, Python's dynamic typing allows variables to change types at runtime, reducing the boilerplate code required in statically typed languages. Memory management is handled by MicroPython's internal garbage collector, which automatically reclaims memory occupied by objects that are no longer in use — preventing memory leaks that frequently plague novice C++ roboticists.

General Purpose Input/Output (GPIO)

At the lowest hardware level, digital control is achieved through General Purpose Input/Output (GPIO) pins. The ESP32 can configure these pins as inputs (to read external voltages) or outputs (to source or sink current to external components). In Python, this hardware abstraction is achieved through the machine.Pin class. Instantiating a pin object (e.g., led = machine.Pin(2, machine.Pin.OUT)) directly manipulates the microcontroller's internal registers to set the pin direction. Methods such as led.on(), led.off(), or led.value(1) allow the Python script to toggle the voltage state between 3.3V (HIGH) and 0V (LOW).

The "SOS" Beacon — Morse Code Flasher

The foundational project introduces digital output control through the creation of an automated Morse code beacon. The hardware consists of a Light Emitting Diode (LED) and an active piezoelectric buzzer wired in parallel to a single GPIO pin configured as a digital output.

Morse code is fundamentally a time-domain protocol. It relies on the precise manipulation of binary states over specific, relative durations. Implementing this in Python requires understanding control flow structures, variable timing, and data structures like dictionaries.

The time.sleep_ms() function is crucial here: it blocks the execution of the Python thread, maintaining the GPIO output state for the exact duration required to emit the distress signal (SOS). This exercise cements the developer's understanding of sequential script execution, the concept of blocking delays, and digital state manipulation.

Lecture 02 · The Body

The Senses — Data Acquisition & Sensors

For an autonomous robot to interact with its physical environment, it must possess mechanisms for data acquisition. Spatial awareness and obstacle avoidance are commonly achieved through time-of-flight (ToF) sensors. The HC-SR04 ultrasonic module serves as the industry standard for educational and prototype robotics due to its simplicity, cost-effectiveness, and reliability within short ranges (2 cm to 400 cm).

Acoustic Transduction & Reflection Physics

The HC-SR04 operates on the physical principles of sonar. The module consists of two primary transducers: a piezoelectric transmitter and a piezoelectric receiver. The Python script commands the ESP32 to send a 10-microsecond HIGH pulse to the sensor's "Trigger" pin. This electrical signal stimulates the piezoelectric crystal, causing it to mechanically deform and emit a burst of eight ultrasonic acoustic pulses at a frequency of 40 kHz. This frequency is utilized because it is well above the threshold of human hearing and offers a highly directional acoustic beam.

When these longitudinal acoustic waves propagate through the air and encounter an object with a significantly different acoustic impedance, a portion of the acoustic energy is reflected back toward the sensor. The receiver crystal detects this echo, transduces the mechanical wave back into an electrical signal, and drives the "Echo" pin HIGH.

The mathematics of determining distance relies on the speed of sound in air, which is approximately 343 meters per second at standard temperature and pressure (20°C), translating to 0.0343 centimeters per microsecond. The fundamental physical formula implemented in the Python script is:

d = ( v × t ) ÷ 2 Where d is the calculated distance to the object, v is the velocity of sound in the medium (0.0343 cm/µs), and t is the total time elapsed between transmission of the pulse and reception of the echo. The division by 2 is essential to account for the two-way travel of the acoustic wave — to the object and back.

Software Implementation & State Logic

To interface with the HC-SR04, the Python code must operate with microsecond precision. MicroPython facilitates this hardware-level timing via the machine.time_pulse_us() function. This function waits for a specified pin to change state (e.g., transitioning from LOW to HIGH) and returns the duration the pin remains in that state, measured in microseconds. This abstracts away the need to manually configure and read hardware timers, showcasing Python's ability to simplify complex embedded operations.

Project Build · L2

Reverse Parking Sensor — Distance Warning System

This project synthesizes sensor data acquisition with conditional Python logic to create an operational state machine. The script continuously polls the ultrasonic sensor inside an infinite while True: loop. The acquired microsecond value is mathematically transformed into a floating-point distance in centimeters.

Using Python's conditional control flow (if, elif, and else statements), the system evaluates the floating-point distance against predefined spatial thresholds:

  • If the calculated distance falls between 50 cm and 149 cm, the state is categorized as safe, and the Python print() function outputs "Perfect!" to the serial console.
  • If the distance falls below the 50 cm threshold, the system triggers an internal warning state, printing "* Too Close *".

This closed-loop feedback mechanism represents the foundational definition of reactivity in robotics: acquire physical data, evaluate that data against software-defined parameters using Python logic, and output a corresponding state change.

Lecture 03 · The Body

Motion — PWM, Servos & DC Locomotion

While digital states are binary (strictly ON or OFF), mechanical systems often require granular, analog-like control for precision positioning. This is achieved through Pulse Width Modulation (PWM) — a technique that dictates the amount of average power delivered to a load by rapidly switching a digital signal HIGH and LOW at a specific frequency.

PWM Fundamentals & Servo Mechanics

A standard hobby servo motor is a complex closed-loop system containing a small DC motor, a reduction gear train, a potentiometer attached to the output shaft for position feedback, and an internal control circuit. Unlike continuous DC motors that spin freely when power is applied, servos are commanded to rotate to specific angles (typically ranging between 0° and 180°).

This angle is dictated by a highly specific PWM signal. Servo motors generally require a control signal operating at a frequency of 50 Hz, which equates to a period of 20 milliseconds per cycle. The physical position of the servo shaft is determined by the duration (width) of the HIGH pulse within that 20-millisecond window.

TBL.2 — Servo pulse width vs. angle (50 Hz)
Command AnglePulse WidthTheoretical Duty Cycle
0° (Minimum)~1.0 ms5%
90° (Neutral / Center)~1.5 ms7.5%
180° (Maximum)~2.0 ms10%

Note: the absolute pulse widths required for full-range motion can vary slightly by manufacturer, sometimes extending from 0.5 ms to 2.5 ms.

MicroPython PWM Implementations

The ESP32 microcontroller features dedicated hardware timers capable of generating PWM signals in the background without blocking the main Python CPU thread. In MicroPython, a PWM object is instantiated and bound to a specific GPIO pin using the machine.PWM(Pin(X)) constructor, followed by setting the frequency using the .freq(50) method.

The duty cycle in MicroPython can be set using several numerical representations depending on the underlying timer resolution. The ESP32 supports 16-bit timer resolutions, allowing the duty cycle to be defined via duty_u16() (accepting values from 0 to 65,535) or by specifying the exact nanosecond pulse width via duty_ns(). When mapping simple integer values like 35, 75, and 115 to servo positions, this implies a specific fractional resolution (often an 8-bit or 10-bit integer mapping used in legacy implementations, where 0–1023 represents 0–100% duty cycle). For example, in a 10-bit system, a 1 ms pulse (5% duty cycle) corresponds to a numerical value of roughly 51 (1023 × 0.05), and a 2 ms pulse corresponds to 102. The values 35, 75, and 115 represent a tuned operational range to push the specific hardware to 0°, 90°, and 180° respectively.

This project constructs a physical security barrier, demonstrating precision analog control. The Python script utilizes a finite for loop to increment the duty cycle variables smoothly, achieving a slow, sweeping motion rather than an abrupt, violent snap to the target position. Locomotion — DC Motors & the H-Bridge Problem

To achieve unconstrained locomotion, an autonomous robot must utilize continuous direct current (DC) motors. However, microcontrollers cannot directly drive DC motors. Motors require significantly more current (often several hundred milliamperes to several amperes) than a standard GPIO pin can source, which is typically capped at a maximum of 12 to 40 mA. Furthermore, inductive loads like motors generate reverse voltage spikes (back Electromotive Force, or EMF) when the magnetic field collapses, which will instantly destroy the sensitive logic gates of a microprocessor.

The L298N Dual H-Bridge Motor Driver

The standard solution to this electrical isolation and current amplification problem is a motor driver. The L298N is an industry-standard integrated circuit featuring two independent H-bridges. An H-bridge is a specialized topological circuit containing four solid-state switching elements (in the L298N, these are Bipolar Junction Transistors, or BJTs) configured in an "H" shape around the motor load. By selectively closing diagonally opposed switches via Python logic signals, the direction of the electrical current flowing through the motor can be reversed — thereby reversing the motor's physical rotational direction.

It is important to note the hardware limitations of the L298N. Because it utilizes older BJT technology rather than modern MOSFETs, it inherently drops approximately 2V across the internal switching transistors. Therefore, a 12V power supply will only deliver an effective 10V to the motors. The L298N module provides control over two motors simultaneously (Motor A and Motor B). The logical control is dictated by six pins connected to the ESP32:

TBL.3 — L298N control pin map
Control PinFunctionTarget Connection
IN1 & IN2Direction logic — Motor AESP32 GPIO (digital output)
IN3 & IN4Direction logic — Motor BESP32 GPIO (digital output)
ENASpeed control — Motor AESP32 GPIO (PWM output)
ENBSpeed control — Motor BESP32 GPIO (PWM output)

If IN1 is driven LOW and IN2 is driven HIGH, the motor rotates forward. If IN1 is HIGH and IN2 is LOW, it rotates backward. If both are LOW, the motor coasts to a stop. If both are HIGH, the H-bridge actively shorts the motor terminals, utilizing back-EMF to brake the motor sharply.

Object-Oriented Python (OOP) for Hardware Abstraction

Managing six independent GPIO pins and two PWM timers sequentially within a main script is prone to error, lacks modularity, and violates the principles of clean code. Python's Object-Oriented Programming (OOP) capabilities allow developers to encapsulate these complex hardware interactions into logical, reusable objects.

By defining a DCMotor Python class, the developer encapsulates the initialization of pins, the setup of PWM frequencies, and the logical H-bridge states. The class constructor method (def __init__(self, pin1, pin2, enable_pin):) accepts the specific pin numbers as parameters. Methods such as .forward(speed) abstract the underlying IN1.value(0), IN2.value(1), and ENA.duty_u16(speed) commands into a single, highly readable Python function call. This demonstrates the power of Python to abstract complex electronic signaling into semantic, human-readable commands.

The Brain I — LLMs & Prompt Engineering

Large Language Models, such as Google's Gemini, are sophisticated neural architectures based on the Transformer model, trained on massive datasets of human text, code, and structured data. Fundamentally, they operate by predicting the next most likely token based on the provided input context. To utilize an LLM as a logical controller for a physical robot, the model cannot be allowed to act as a free-form chatbot; it must be rigidly constrained and programmed using precise natural language instructions — a discipline known as prompt engineering.

The 3-Layer Prompt Hierarchy

To construct a reliable robotic cognitive engine, prompts must adhere to a strict architectural hierarchy:

  1. Master / System Prompt (System Instructions): the persistent digital identity and overarching rule set of the AI. It dictates the model's fundamental behavior, operational boundaries, and output format. In the Gemini API, this is passed via the systemInstruction parameter, ensuring the model never forgets its primary directive, even as the conversation length grows.
  2. Context / Standard Prompts: the iterative, dynamic inputs provided to the model during operation. In this framework, the standard prompt consists of the real-time telemetry data gathered from the ultrasonic sensor.
  3. Few-Shot Examples: providing the model with structural examples of the exact input-output pairings expected, cementing the JSON formatting rules and preventing hallucination.

Model Parameters — Controlling Probabilistic Output

LLM outputs are inherently probabilistic. When generating a response, the model calculates a probability distribution over its entire vocabulary. Developers can manipulate this distribution using specific parameters within the API payload:

  • Temperature: this parameter scales the internal logits before they are passed through the softmax function. A low temperature (e.g., 0.1 to 0.2) steepens the distribution, forcing the model to almost exclusively select the highest-probability token. This yields highly deterministic, analytical outputs — strictly required for generating code or robotic commands. Conversely, a high temperature (e.g., 0.8+) flattens the curve, allowing lower-probability tokens to be selected, yielding creative and varied text.
  • Top-P (Nucleus Sampling): this parameter restricts token selection to a subset of tokens whose cumulative probability equals the Top-P value. For a navigational agent, setting Top-P low (e.g., 0.1) ensures the model only considers the most structurally sound responses.
Project Build · L4

Persona Architect — System Prompt Design

This project requires drafting the system instructions for the robot using a highly constrained prompt formula: [Role] + [Context] + [Task Detail] + [Negative Constraints] + [Desired Format].

Negative constraints are the most critical element in AI robotics. LLMs have an inherent tendency to "hallucinate" conversational filler (e.g., "Sure, I can help with that! Based on the sensor data, here is your JSON:"). This extraneous text immediately breaks algorithmic JSON parsing on the ESP32. By explicitly utilizing negative constraints ("DO NOT provide conversational text. DO NOT explain your reasoning. ONLY output a valid JSON object"), the developer shapes the probabilistic LLM into a precise, deterministic software function suitable for machine-to-machine communication.

Lecture 05 · The Brain

The Bridge & The Connection — APIs, Postman & the ESP32 Client

To connect the local microcontroller to the cloud-based AI, the system must traverse the internet utilizing standardized communication protocols. The Google Gemini API facilitates this via Representational State Transfer (REST) endpoints.

REST APIs & HTTP Protocols

REST APIs utilize the Hypertext Transfer Protocol (HTTP) over TCP/IP sockets. When a client (the robot) requires data processing from Gemini, it formulates an HTTP POST request. The POST request encapsulates the required data within its body and sends it to a specific URL endpoint (e.g., https://generativelanguage.googleapis.com/v1beta/models/gemini-1.5-flash:generateContent).

A successful API transaction requires precise header management. The HTTP headers must specify the Content-Type as application/json to inform the server of the payload format. Furthermore, authentication credentials must be provided — either via a specific header like x-goog-api-key or appended directly to the URL string as a query parameter (?key=YOUR_API_KEY) — authorizing the transaction.

JSON Payload Architecture

Data transmitted to and from the Gemini API is formatted exclusively in JSON (JavaScript Object Notation). JSON is a lightweight, human-readable data-interchange format built on two universal structures: collections of name/value pairs (analogous to Python dictionaries) and ordered lists of values (analogous to Python lists).

A request payload sent to the Gemini API requires a highly specific hierarchical schema. It contains a root contents array, which holds parts objects containing the text prompts. Furthermore, the systemInstruction object and the generation_config object (where parameters like temperature and max_output_tokens are strictly defined) are declared at the root level of the JSON payload.

Project Build · L5-A

The API Inspector — Manual Intelligence Testing

Before writing embedded Python code on the ESP32, the developer must validate the API interaction manually to avoid debugging network and logic errors simultaneously. Using Postman — a dedicated API development and testing environment — the developer constructs the exact POST request payload. By injecting mock sensor data into the payload and executing the request against the gemini-1.5-flash model, the developer inspects the raw JSON response returned by Google's servers.

Analyzing the response structure is paramount. The Gemini API returns a complex, nested JSON object where the actual text response generated by the model is buried several layers deep — specifically located at response["candidates"][0]["content"]["parts"][0]["text"]. Understanding this exact hierarchical path is an absolute prerequisite for writing the subsequent dictionary parsing logic in MicroPython.

MicroPython Network Configuration

Transitioning the cloud intelligence to the physical robot requires executing HTTP requests natively on the ESP32. This represents one of the most resource-intensive operations for a microcontroller, demanding careful handling of network interfaces, memory management, and data parsing constraints.

The ESP32 must first connect to a local Wi-Fi router to gain internet access. In MicroPython, the network module is utilized to instantiate a Wireless Local Area Network (WLAN) interface configured in Station Mode (network.STA_IF). Proper network management requires differentiating between the ESP32's two primary boot scripts:

  • boot.py: executed immediately upon power-up or hard reset. It is an embedded best practice to place the Wi-Fi connection logic here. By executing wlan.connect(ssid, password) and waiting for wlan.isconnected() to return True, the board establishes network access before the main application runs.
  • main.py: runs automatically after boot.py completes and contains the primary application loop. Separating these concerns ensures that programmatic soft resets (machine.soft_reset()) used for error recovery do not needlessly drop and renegotiate the Wi-Fi connection, saving valuable boot time.

Executing HTTP Requests & Serializing Data

To send data to the Gemini API, MicroPython utilizes the urequests (micro-requests) module — a subset of the standard Python requests library optimized for constrained memory environments. To execute a POST request, the script first constructs a Python dictionary mirroring the required Gemini JSON schema.

However, a common failure point in embedded Python is attempting to pass a dictionary directly to urequests.post(), which yields a TypeError: object with buffer protocol required. The HTTP standard requires payloads to be transmitted as octet streams (bytes or strings). MicroPython provides the ujson module for this exact task: the ujson.dumps() (dump string) function serializes the hierarchical Python dictionary into a linear, UTF-8 encoded JSON string suitable for transmission over the TCP/IP socket.

Memory Management & Parsing

When the API returns a response, the ESP32 receives a JSON-formatted text string. The ujson.loads() (load string) function deserializes this text back into a standard Python dictionary, allowing the developer to access the nested data using standard key-index notation.

Because the ESP32 possesses limited heap memory, handling unexpectedly large JSON strings from LLMs can easily trigger a MemoryError. Efficient embedded Python coding practices are vital. This includes configuring the API's max_output_tokens parameter to strictly bound the response size, and explicitly calling Python's garbage collector (import gc; gc.collect()) after the HTTP response object is parsed and closed (response.close()) to reclaim memory for the next loop iteration.

Lecture 06 · Capstone

Capstone — The AI-Commanded Robot

The culmination of the curriculum is the creation of a closed-loop cyber-physical system. The "Sentient Explorer" project synthesizes every prior concept: acoustic physics, PWM motor control, Object-Oriented Python, embedded networking, and probabilistic LLM reasoning.

The Cyber-Physical Architecture

In a closed-loop control system, output commands are continually adjusted based on incoming physical sensor feedback. For this robot, the continuous operational loop is orchestrated by the main Python script as follows:

  1. Acquisition: the HC-SR04 ultrasonic sensor acquires raw microsecond data, which Python mathematical operations convert into a physical distance metric.
  2. Aggregation: the ESP32 utilizes Python f-strings to format this distance metric into a structured prompt context (e.g., f"Sensor reading: Obstacle detected at {distance} cm. What is the optimal maneuver?").
  3. Transmission: the prompt is structured into a dictionary, serialized via ujson.dumps(), and transmitted to the Gemini API using urequests.post().
  4. Cognition: the gemini-1.5-flash model evaluates the prompt against its Persona Architect system instructions in the cloud. It applies spatial reasoning to determine that, for instance, a 15 cm reading represents an impending collision requiring evasive action.
  5. Response: the LLM generates a strictly formatted JSON response (forced by negative constraints and a low temperature parameter) containing a command directive (e.g., {"command": "reverse_right"}).
  6. Actuation: the ESP32 receives the JSON, parses the command string, and routes the logic using Python if/elif statements to the corresponding DCMotor OOP class methods. The L298N driver reverses the polarity of the H-bridge on one side, physically executing the pivot.

Fault Tolerance, Exception Handling & Fail-Safes

Relying on cloud AI for real-time motor control introduces severe vulnerabilities — primarily network latency and the risk of API timeouts. A robust robotic system must anticipate and gracefully handle failure. The final Python implementation mandates the use of try...except blocks around all network requests.

If the urequests.post() operation takes longer than a predefined timeout period, or if the Wi-Fi connection drops unexpectedly, the exception handler catches the error. Crucially, the exception block must immediately force the robot into a physically safe state by executing a .stop() method, setting all motor PWM duty cycles to zero to prevent runaway collisions. Furthermore, to recover from irrecoverable memory states, socket lockups, or continuous connection failures, the machine.reset() or machine.soft_reset() commands can be programmatically triggered within the exception handler to reboot the microcontroller cleanly.
